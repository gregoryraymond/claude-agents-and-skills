# Mesh Decimation via Quadric Error Metrics

Reference skill for implementing automatic mesh simplification to replace or augment the current fixed 2-level LOD system (`coastline_lod0.glb` at 2048x1560, `coastline_lod1.glb` at 4096x3120) with continuous or multi-level LOD generated by QEM-based edge collapse.

## 1. Quadric Error Metrics (QEM) -- Garland & Heckbert 1997

The foundational algorithm from "Surface Simplification Using Quadric Error Metrics" (SIGGRAPH 97).

### Core Idea

Each vertex accumulates a **quadric matrix** (4x4 symmetric matrix) that encodes the squared distance from that vertex to all planes of its adjacent triangles. Collapsing an edge merges two vertices; the cost of the collapse equals the geometric error introduced, computed via the merged quadric.

### Quadric Matrix Construction

For each triangle with plane equation `ax + by + cz + d = 0` (where `[a,b,c]` is the unit normal, `d = -dot(normal, point_on_plane)`):

```
        [ a*a  a*b  a*c  a*d ]
K_p =   [ a*b  b*b  b*c  b*d ]
        [ a*c  b*c  c*c  c*d ]
        [ a*d  b*d  c*d  d*d ]
```

The quadric `Q_v` for vertex `v` is the sum of `K_p` for all triangles adjacent to `v`:

```
Q_v = sum(K_p for each triangle touching v)
```

The error of placing a vertex at position `v = [x, y, z, 1]^T` is:

```
error(v) = v^T * Q_v * v
```

This is a scalar: the sum of squared distances from `v` to all original planes.

### Edge Collapse Cost

For edge `(v1, v2)`, the merged quadric is:

```
Q_merged = Q_v1 + Q_v2
```

The **optimal position** for the merged vertex minimizes `v^T * Q_merged * v`. This is found by solving:

```
    [ q11  q12  q13  q14 ] [ x ]   [ 0 ]
    [ q12  q22  q23  q24 ] [ y ] = [ 0 ]
    [ q13  q23  q33  q34 ] [ z ]   [ 0 ]
    [  0    0    0    1  ] [ 1 ]   [ 1 ]
```

If the upper-left 3x3 submatrix is invertible, solve for `[x, y, z]`. Otherwise, fall back to:
1. Midpoint of `v1` and `v2`
2. Whichever of `v1`, `v2` gives lower error

The **cost** of this edge collapse is `error(v_optimal) = v_optimal^T * Q_merged * v_optimal`.

### Why QEM Works Well

- **O(1) per collapse**: Each collapse only needs the two endpoint quadrics (constant-size 4x4 matrices).
- **Global error tracking**: The quadric accumulates error from the original surface planes, so even after many collapses, the error metric reflects deviation from the *original* mesh, not just the previous step.
- **Attribute preservation**: Extended QEM (Garland 1998) can incorporate vertex colors, UVs, and normals into the quadric by expanding to higher-dimensional error metrics.

## 2. Edge Collapse Operation

An edge collapse `(v1, v2) -> v_new` involves:

### Topological Changes

1. **Remove** the edge `(v1, v2)` and the 1-2 triangles sharing it (the "ear" triangles).
2. **Merge** `v1` and `v2` into `v_new` at the optimal position.
3. **Retarget** all edges and triangles that referenced `v2` to point to `v_new` instead.
4. **Remove degenerate triangles**: After retargeting, some triangles may have two identical vertices -- remove them.
5. **Update adjacency**: All neighbor lists, half-edges, and face lists must reflect the merge.

### Validity Checks (Critical)

Before performing a collapse, verify:

- **No mesh inversion**: The collapse must not flip any adjacent triangle's normal. Check that `dot(old_normal, new_normal) > 0` for all affected triangles.
- **No non-manifold creation**: The collapse must not create non-manifold edges (edges shared by more than 2 triangles) or non-manifold vertices.
- **Link condition**: The intersection of the 1-ring neighborhoods of `v1` and `v2` must equal exactly the vertices of the shared triangles. If violated, the collapse creates a topological defect.

### Attribute Interpolation

When merging vertices with different vertex colors, UVs, or normals:
- **Vertex colors**: Weighted average based on distance from optimal position to original positions, or use the extended QEM that minimizes color error jointly.
- **UVs**: Must be careful -- UV discontinuities (seams) should prevent collapse across seam edges.
- **Normals**: Recompute from the new triangle configuration after collapse.

For the terrain mesh specifically: vertex colors encode cliff/beach/satellite blend factors (the `alpha` channel controls terrain shader blending). These MUST be interpolated correctly or the coastal rendering breaks.

## 3. Priority Queue

### Structure

A **min-heap** (binary heap or Fibonacci heap) of all valid edges, keyed by collapse cost:

```rust
struct CollapseCandidate {
    edge: EdgeId,
    cost: f64,
    optimal_pos: Vec3,
    // Generation counter to detect stale entries
    generation: u32,
}
```

### Lazy Deletion Pattern

When an edge collapse modifies neighboring edges' costs, rather than updating the heap in-place (expensive), use lazy deletion:

1. Each edge has a `generation` counter.
2. When an edge's neighborhood changes, increment its generation and push a new entry with updated cost.
3. When popping from the heap, skip entries whose generation doesn't match the edge's current generation.

This avoids O(log n) decrease-key operations and works well with Rust's `BinaryHeap` (which doesn't support decrease-key).

### Algorithm Loop

```
1. Compute Q_v for every vertex
2. For every edge (v1, v2):
   a. Compute Q_merged = Q_v1 + Q_v2
   b. Find optimal position v_opt
   c. Compute cost = v_opt^T * Q_merged * v_opt
   d. Insert into priority queue
3. While (triangle_count > target) and (queue not empty):
   a. Pop minimum-cost edge
   b. Skip if stale (generation mismatch)
   c. Skip if collapse would be invalid (normal flip, non-manifold)
   d. Perform collapse: merge vertices, remove triangles
   e. Update Q_v_new = Q_v1 + Q_v2
   f. For each edge incident to v_new:
      - Recompute cost with new quadric
      - Push updated entry to queue (increment generation)
4. Output simplified mesh
```

### Complexity

- Initial quadric computation: O(V) where V = vertex count
- Initial edge costs: O(E) where E = edge count
- Each collapse: O(degree) for neighborhood update
- Total: O(E log E) for the full simplification

For the terrain mesh at LOD 0 (2048x1560 = ~3.2M vertices, ~6.4M triangles), expect the full QEM pass to take 5-30 seconds in optimized Rust, depending on target reduction.

## 4. Boundary Preservation

### Why Boundaries Matter for Terrain

The terrain mesh has critical boundary types:
1. **Coastline edges**: Land-ocean transitions where cliff walls connect. These define the map's visual identity.
2. **Mesh perimeter**: The outer edges of the Europe terrain rectangle.
3. **Beach zone boundaries**: Transitions between beach-sloped and cliff-walled coastline sections.

Simplifying these boundaries too aggressively destroys coastline shape, creates gaps between terrain and cliff walls, and misaligns the territory overlay.

### Boundary Weighting via Penalty Planes

The standard technique: for each boundary edge, add a **penalty plane** perpendicular to the mesh surface and passing through the edge. This plane's quadric is added to both endpoint vertices with a **large weight** (10x-1000x normal triangle weight).

```
For boundary edge (v1, v2):
  edge_vec = v2 - v1
  face_normal = normal of adjacent triangle
  penalty_normal = normalize(cross(edge_vec, face_normal))
  penalty_d = -dot(penalty_normal, v1)
  K_boundary = penalty_weight * outer_product([a, b, c, d])
  Q_v1 += K_boundary
  Q_v2 += K_boundary
```

The penalty planes make it very expensive to move boundary vertices away from their original positions, so the greedy algorithm collapses interior edges first and preserves boundary shape.

### Coastline-Specific Strategy

For the Europe terrain mesh:

| Boundary Type | Penalty Weight | Rationale |
|---|---|---|
| Coastline edges (land-ocean) | 100x-500x | Coastline shape is the most visually important feature |
| Beach zone edges | 50x-200x | Beach slope geometry must match ocean surface intersection |
| Country polygon boundaries | 10x-50x | Territory overlay alignment (less critical -- overlay has its own mesh) |
| Mesh perimeter | 1000x (effectively locked) | Must not move -- connects to ocean/fog meshes |
| Interior edges | 1x (no penalty) | Free to collapse |

### Vertex Locking

For vertices that must NEVER move (e.g., mesh corners, specific control points):

```rust
// Set quadric to a large diagonal matrix
Q_locked = LARGE_VALUE * I_4x4  // Makes any movement infinitely expensive
```

This is equivalent to infinite-weight penalty planes in all directions.

## 5. Terrain-Specific Considerations

### Heightmap Peak/Valley Preservation

Flat terrain (plains, plateaus) should lose triangles aggressively. Peaks, ridges, and valleys must retain detail. QEM handles this naturally because:

- Flat regions: All triangle planes are nearly parallel, so the quadric's eigenvalues are small in the flat directions. Collapsing edges in flat areas costs almost nothing.
- Peaks/ridges: Triangle planes converge from multiple directions, creating a quadric with large eigenvalues. Moving the vertex away from the peak costs a lot.

However, for subtle elevation changes (gentle hills), the default QEM may simplify too aggressively. Add **curvature weighting**:

```rust
// Weight each triangle's contribution by local curvature
let curvature = estimate_gaussian_curvature(vertex);
let weight = 1.0 + curvature_scale * curvature.abs();
Q_v += weight * K_triangle;
```

### Vertex Color Preservation (Critical for This Project)

The terrain mesh uses vertex colors for shader control:
- **RGB**: Warm tint for coastal masking (hides blue satellite bleeding)
- **Alpha**: Controls cliff texture vs satellite texture blend in the terrain shader (`smoothstep(0.85, 0.45, alpha)`)

Collapsing an edge that joins a coastal vertex (alpha=0.45) with an inland vertex (alpha=1.0) and placing the result at the midpoint would create a vertex with alpha=0.725, which the shader interprets very differently.

**Solution**: Use extended QEM that incorporates vertex attributes into the error metric. For each vertex, expand the position from `[x, y, z, 1]` to `[x, y, z, r, g, b, a, 1]` and compute 8x8 quadrics. This jointly minimizes geometric error AND attribute error.

Alternatively (simpler): add an attribute-difference penalty to edge collapse cost:

```rust
let attr_cost = attribute_weight * (
    (v1.color - v2.color).length_squared()
    + (v1.alpha - v2.alpha).powi(2) * alpha_weight
);
let total_cost = geometric_cost + attr_cost;
```

This prevents collapsing edges that span large attribute discontinuities (e.g., the coastal mask boundary).

### Coastline Map Preservation

The terrain has an associated `coastline_map_lodN.bin` that maps each grid cell to a country index. After QEM simplification, this per-cell map no longer aligns with the simplified mesh vertices. Solutions:

1. **Bake country ID into vertex attributes** before simplification, then use barycentric interpolation (round to nearest integer) after.
2. **Keep the map at original resolution** and sample it using the simplified mesh's UV coordinates.
3. **Generate a new map** by rasterizing the simplified mesh triangles with their inherited country IDs.

Option 2 is simplest and works if UV coordinates are preserved through simplification.

### Island Preservation

Small islands (Crete, Corsica, Sardinia, Cyprus) may be reduced to too few triangles by aggressive simplification. The current code already has island protection logic (ratio-based, scaling with grid resolution). For QEM:

- **Pre-tag island vertices** with high boundary weights so their coastlines resist simplification.
- **Set per-island minimum triangle counts** based on island area.
- **Or**: exclude island vertices from simplification entirely and only decimate the mainland.

## 6. Progressive Meshes

### Concept (Hoppe 1996)

A progressive mesh stores the mesh as:
1. A **base mesh** (coarsest level, M triangles)
2. A sequence of **vertex split records** that incrementally refine it back to the original

Each vertex split is the inverse of an edge collapse:

```
Vertex Split Record:
  - vertex_id: which vertex to split
  - new_vertex_pos: position of the new vertex
  - new_vertex_attrs: colors, normals, UVs
  - affected_triangles: which triangles change connectivity
```

### Continuous LOD

At runtime, apply vertex splits or edge collapses to reach the desired detail level:

```
Full detail mesh (N triangles)
  <- vertex split N-1
  <- vertex split N-2
  ...
  <- vertex split M+1
Base mesh (M triangles)
```

Any intermediate state is a valid mesh. The runtime picks a triangle budget based on camera distance and applies/reverses splits to hit it.

### View-Dependent Progressive Meshes

Rather than a single global LOD level, different parts of the mesh can be at different detail levels:

- **Close to camera**: Full detail (all splits applied)
- **Far from camera**: Coarse (many splits reversed)
- **Coastlines in view**: Keep high detail regardless of distance
- **Off-screen regions**: Minimal detail

This is the most sophisticated approach and requires a runtime system that evaluates per-vertex screen-space error to decide which splits to apply.

### Geomorphing

To avoid "popping" when LOD changes, interpolate vertex positions between the pre-split and post-split positions over several frames:

```rust
let t = smoothstep(lod_transition_progress);
let pos = lerp(coarse_pos, fine_pos, t);
```

This produces smooth visual transitions instead of sudden geometry changes.

## 7. Adaptive Detail

### Error-Driven Simplification

Rather than targeting a fixed triangle count, set a **maximum screen-space error threshold** (e.g., 0.5 pixels). The simplification continues collapsing edges until no remaining collapse would introduce more than the threshold error when rendered.

This naturally produces:
- **More triangles** at complex features (coastlines, mountain ridges)
- **Fewer triangles** in flat areas (plains, ocean floor)
- **Detail proportional to visual importance**

### Multiple LOD Levels from One Pass

Run QEM simplification once, recording the collapse sequence. Then extract meshes at multiple checkpoints:

```
Original:     6.4M triangles (LOD 3 -- maximum zoom)
After 50% :   3.2M triangles (LOD 2 -- close zoom)
After 75% :   1.6M triangles (LOD 1 -- medium zoom)
After 90% :   640K triangles (LOD 0 -- full map view)
After 95% :   320K triangles (LOD -1 -- minimap)
```

Each level is a valid, high-quality simplification because QEM preserves the most important features longest.

### Spatial Partitioning for Hybrid LOD

Divide the terrain into chunks (e.g., 16x16 grid). Each chunk has its own LOD level based on camera distance. Chunks at different LODs must have **matching boundary vertices** to avoid cracks:

- **Restricted boundary collapse**: Never collapse edges on chunk boundaries.
- **T-junction stitching**: Insert degenerate triangles at LOD boundaries to bridge different vertex densities.
- **Geomorphing at boundaries**: Blend vertex positions at chunk edges.

## 8. Comparison with Current Fixed LOD System

### Current System (in `gen_cliff_glb.rs`)

```
LOD 0: 2048 x 1560 grid -> ~6.4M triangles (zoomed out, always loaded)
LOD 1: 4096 x 3120 grid -> ~25.6M triangles (zoomed in, lazy loaded)
Switch at camera distance 32-38 units (with hysteresis)
```

**Pros of current approach:**
- Simple, predictable
- No runtime mesh modification
- Grid alignment makes country map lookup trivial
- Beach/cliff/coastal logic operates on a regular grid

**Cons:**
- Only 2 levels: either too many or too few triangles
- LOD 1 is 4x the triangles of LOD 0 -- large jump, wastes GPU for partially-zoomed views
- Flat inland areas have the same vertex density as complex coastlines
- LOD 1 uses ~360 MB GPU memory
- No smooth transition (pop when switching)

### QEM-Based System (proposed)

**Pros:**
- Automatic detail distribution: coastlines get 10x more triangles than plains
- Multiple LOD levels from one simplification pass
- Progressive mesh enables smooth transitions (no pop)
- Can reduce total triangle count while IMPROVING coastline quality
- Memory-efficient: each LOD level is smaller, and progressive mesh only stores deltas

**Cons:**
- More complex offline generation (but still offline, not runtime)
- Country map alignment requires re-baking or UV-based lookup
- Cliff wall geometry must be updated to match simplified coastline edges
- Beach/coastal vertex color logic must be applied BEFORE simplification
- Testing is harder -- must verify coastline quality at all LOD levels

### Migration Path

1. **Phase 1**: Apply QEM to LOD 0 to produce 3-4 fixed LOD levels with adaptive detail. Keep the regular grid generation but simplify as a post-process. Validate coastline quality.
2. **Phase 2**: Integrate progressive mesh for smooth LOD transitions at runtime. Replace the binary LOD switch in `terrain_lod_system`.
3. **Phase 3**: View-dependent LOD with per-chunk detail levels for maximum efficiency.

## 9. Implementation in Rust

### Data Structure: Index-Based Half-Edge Mesh

Avoid `Rc<RefCell<>>` graph structures (slow, cache-unfriendly). Use index-based storage with arena allocation:

```rust
/// Compact half-edge mesh using index-based references.
/// All data stored in flat Vec arrays for cache efficiency.

type VertexId = u32;
type HalfEdgeId = u32;
type FaceId = u32;

struct Vertex {
    position: [f64; 3],
    color: [f32; 4],        // RGBA vertex color (terrain shader control)
    halfedge: HalfEdgeId,   // One outgoing half-edge
    quadric: Quadric,       // 4x4 symmetric matrix (10 unique floats)
    generation: u32,        // For lazy deletion in priority queue
    is_boundary: bool,
    is_removed: bool,
}

struct HalfEdge {
    origin: VertexId,       // Vertex this half-edge starts from
    twin: HalfEdgeId,       // Opposite half-edge (same edge, other direction)
    next: HalfEdgeId,       // Next half-edge in the face loop
    prev: HalfEdgeId,       // Previous half-edge in the face loop
    face: FaceId,           // Face to the left of this half-edge
    is_removed: bool,
}

struct Face {
    halfedge: HalfEdgeId,   // One half-edge on this face's boundary
    is_removed: bool,
}

/// 4x4 symmetric matrix stored as 10 unique values.
/// Q = [[ q0  q1  q2  q3 ]
///      [ q1  q4  q5  q6 ]
///      [ q2  q5  q7  q8 ]
///      [ q3  q6  q8  q9 ]]
#[derive(Clone, Copy, Default)]
struct Quadric {
    data: [f64; 10],
}

struct HalfEdgeMesh {
    vertices: Vec<Vertex>,
    halfedges: Vec<HalfEdge>,
    faces: Vec<Face>,
    // Free lists for reuse after removal
    free_vertices: Vec<VertexId>,
    free_halfedges: Vec<HalfEdgeId>,
    free_faces: Vec<FaceId>,
}
```

### Why Index-Based (Not Pointer-Based)

- **No `Rc<RefCell<>>`**: Rust's borrow checker makes pointer-based graph structures painful. Index-based is idiomatic.
- **Cache friendly**: Flat arrays mean sequential access patterns are fast.
- **Serializable**: Can save/load the mesh state trivially.
- **The `half_edge_mesh` crate** exists but uses `Weak<RefCell<>>` which is ~3-5x slower than index-based for mesh operations.

### Quadric Operations

```rust
impl Quadric {
    /// Create quadric from plane equation ax + by + cz + d = 0
    fn from_plane(a: f64, b: f64, c: f64, d: f64) -> Self {
        Quadric {
            data: [
                a*a, a*b, a*c, a*d,
                     b*b, b*c, b*d,
                          c*c, c*d,
                               d*d,
            ],
        }
    }

    /// Add two quadrics
    fn add(&self, other: &Quadric) -> Quadric {
        let mut result = Quadric::default();
        for i in 0..10 {
            result.data[i] = self.data[i] + other.data[i];
        }
        result
    }

    /// Evaluate error: v^T * Q * v where v = [x, y, z, 1]
    fn evaluate(&self, x: f64, y: f64, z: f64) -> f64 {
        let q = &self.data;
        // Expand v^T * Q * v for v = [x, y, z, 1]
        q[0]*x*x + 2.0*q[1]*x*y + 2.0*q[2]*x*z + 2.0*q[3]*x
                 +     q[4]*y*y + 2.0*q[5]*y*z + 2.0*q[6]*y
                                +     q[7]*z*z + 2.0*q[8]*z
                                               +     q[9]
    }

    /// Find position minimizing v^T * Q * v.
    /// Returns None if the 3x3 submatrix is singular.
    fn optimal_position(&self) -> Option<[f64; 3]> {
        let q = &self.data;
        // Solve the 3x3 system:
        // [ q0  q1  q2 ] [ x ]   [ -q3 ]
        // [ q1  q4  q5 ] [ y ] = [ -q6 ]
        // [ q2  q5  q7 ] [ z ]   [ -q8 ]
        let a = [[q[0], q[1], q[2]],
                  [q[1], q[4], q[5]],
                  [q[2], q[5], q[7]]];
        let b = [-q[3], -q[6], -q[8]];

        let det = a[0][0] * (a[1][1]*a[2][2] - a[1][2]*a[2][1])
                - a[0][1] * (a[1][0]*a[2][2] - a[1][2]*a[2][0])
                + a[0][2] * (a[1][0]*a[2][1] - a[1][1]*a[2][0]);

        if det.abs() < 1e-10 {
            return None;
        }

        let inv_det = 1.0 / det;
        let x = inv_det * (b[0] * (a[1][1]*a[2][2] - a[1][2]*a[2][1])
                         - b[1] * (a[0][1]*a[2][2] - a[0][2]*a[2][1])
                         + b[2] * (a[0][1]*a[1][2] - a[0][2]*a[1][1]));
        let y = inv_det * (a[0][0] * (b[1]*a[2][2] - b[2]*a[2][1])
                         - a[1][0] * (b[0]*a[2][2] - b[2]*a[2][0])
                         + a[2][0] * (b[0]*a[2][1] - b[1]*a[2][0])); // Cramer's rule, middle column
        let z = inv_det * (a[0][0] * (a[1][1]*b[2] - a[1][2]*b[1])
                         - a[0][1] * (a[1][0]*b[2] - a[1][2]*b[0])
                         + a[0][2] * (a[1][0]*b[1] - a[1][1]*b[0]));

        Some([x, y, z])
    }

    /// Scale quadric by a weight factor (for boundary penalties)
    fn scale(&self, w: f64) -> Quadric {
        let mut result = Quadric::default();
        for i in 0..10 {
            result.data[i] = self.data[i] * w;
        }
        result
    }
}
```

### Priority Queue with Lazy Deletion

```rust
use std::cmp::Ordering;
use std::collections::BinaryHeap;

struct CollapseCandidate {
    cost: f64,
    edge_id: HalfEdgeId,  // Use the lower-indexed half-edge of the pair
    generation: u32,
}

// Reverse ordering for min-heap (BinaryHeap is a max-heap)
impl Ord for CollapseCandidate {
    fn cmp(&self, other: &Self) -> Ordering {
        other.cost.partial_cmp(&self.cost)
            .unwrap_or(Ordering::Equal)
            .then_with(|| self.edge_id.cmp(&other.edge_id))
    }
}

struct SimplificationQueue {
    heap: BinaryHeap<CollapseCandidate>,
    // Per-edge generation counters (indexed by half-edge id)
    edge_generations: Vec<u32>,
}

impl SimplificationQueue {
    fn push(&mut self, edge_id: HalfEdgeId, cost: f64) {
        self.edge_generations[edge_id as usize] += 1;
        self.heap.push(CollapseCandidate {
            cost,
            edge_id,
            generation: self.edge_generations[edge_id as usize],
        });
    }

    fn pop_valid(&mut self) -> Option<CollapseCandidate> {
        while let Some(candidate) = self.heap.pop() {
            if candidate.generation == self.edge_generations[candidate.edge_id as usize] {
                return Some(candidate);
            }
            // Stale entry -- skip
        }
        None
    }
}
```

### Main Simplification Loop

```rust
fn simplify_mesh(mesh: &mut HalfEdgeMesh, target_faces: usize) {
    // 1. Compute initial quadrics
    for face_id in 0..mesh.faces.len() {
        if mesh.faces[face_id].is_removed { continue; }
        let (a, b, c, d) = mesh.face_plane(face_id as FaceId);
        let kp = Quadric::from_plane(a, b, c, d);
        // Add to each vertex of this face
        for vid in mesh.face_vertices(face_id as FaceId) {
            mesh.vertices[vid as usize].quadric =
                mesh.vertices[vid as usize].quadric.add(&kp);
        }
    }

    // 2. Add boundary penalty planes
    for he_id in 0..mesh.halfedges.len() {
        if !mesh.is_boundary_edge(he_id as HalfEdgeId) { continue; }
        let (v1, v2) = mesh.edge_vertices(he_id as HalfEdgeId);
        let weight = classify_boundary_weight(mesh, he_id as HalfEdgeId);
        let penalty = compute_boundary_penalty(mesh, he_id as HalfEdgeId);
        let kp = penalty.scale(weight);
        mesh.vertices[v1 as usize].quadric =
            mesh.vertices[v1 as usize].quadric.add(&kp);
        mesh.vertices[v2 as usize].quadric =
            mesh.vertices[v2 as usize].quadric.add(&kp);
    }

    // 3. Initialize priority queue
    let mut queue = SimplificationQueue::new(mesh.halfedges.len());
    for he_id in 0..mesh.halfedges.len() {
        if mesh.halfedges[he_id].is_removed { continue; }
        // Only process one half-edge per edge pair
        if he_id > mesh.halfedges[he_id].twin as usize { continue; }
        let cost = compute_collapse_cost(mesh, he_id as HalfEdgeId);
        queue.push(he_id as HalfEdgeId, cost);
    }

    // 4. Collapse loop
    let mut face_count = mesh.active_face_count();
    while face_count > target_faces {
        let candidate = match queue.pop_valid() {
            Some(c) => c,
            None => break,
        };

        if !validate_collapse(mesh, candidate.edge_id) {
            continue;
        }

        let affected_edges = perform_collapse(mesh, candidate.edge_id);
        face_count -= 2; // Each interior collapse removes 2 faces

        // Update costs for affected edges
        for &edge_id in &affected_edges {
            let cost = compute_collapse_cost(mesh, edge_id);
            queue.push(edge_id, cost);
        }
    }
}
```

### Converting From Regular Grid

The current `gen_cliff_glb.rs` outputs a regular grid mesh. To apply QEM:

```rust
fn grid_to_halfedge_mesh(
    positions: &[[f32; 3]],
    indices: &[u32],
    colors: &[[f32; 4]],
) -> HalfEdgeMesh {
    // 1. Add all vertices
    // 2. For each triangle (3 consecutive indices):
    //    a. Create 3 half-edges forming a face loop
    //    b. Link next/prev pointers within the face
    // 3. For each pair of opposite half-edges, link twins
```

## 10. Replacing the Current 2-Level LOD System

### Architecture Overview

```
Current:
  gen_cliff_glb.rs -> coastline_lod0.glb (2048x1560 grid, ~6.4M tris)
                   -> coastline_lod1.glb (4096x3120 grid, ~25.6M tris)
  map.rs: binary switch at camera distance 32-38

Proposed:
  gen_cliff_glb.rs -> full_detail.glb (4096x3120 grid, ~25.6M tris)
                   -> QEM simplifier
                      -> coastline_lod3.glb (~25.6M tris, maximum detail)
                      -> coastline_lod2.glb (~6.4M tris, close zoom)
                      -> coastline_lod1.glb (~1.6M tris, medium zoom)
                      -> coastline_lod0.glb (~400K tris, full map view)
                      -> progressive_mesh.bin (collapse sequence for smooth transitions)
  map.rs: continuous LOD via progressive mesh, or multi-level discrete with geomorphing
```

### Integration Points in gen_cliff_glb.rs

The simplification step runs AFTER all vertex colors, beach slopes, cliff walls, and coastal masking are applied. The QEM simplifier receives the final, fully-attributed mesh:

```rust
fn generate_lod(heightmap: &HeightMap, cols: usize, rows: usize, ...) {
    // ... existing terrain generation code ...
    // ... beach slopes, cliff walls, coastal BFS, vertex colors ...

    // NEW: Apply QEM simplification to generate multiple LOD levels
    let full_mesh = build_halfedge_mesh(&positions, &indices, &colors, &normals);

    let lod_targets = [
        (25_600_000, "coastline_lod3.glb"),  // Full detail
        ( 6_400_000, "coastline_lod2.glb"),  // ~75% reduction
        ( 1_600_000, "coastline_lod1.glb"),  // ~94% reduction
        (   400_000, "coastline_lod0.glb"),  // ~98% reduction
    ];

    let mut mesh = full_mesh.clone();
    for &(target_faces, filename) in &lod_targets {
        simplify_mesh(&mut mesh, target_faces);
        export_glb(&mesh, filename);
        // Country map must be regenerated for each LOD
        export_country_map(&mesh, &country_polygons, map_filename);
    }
}
```

### Runtime Changes in map.rs

Replace the binary `TerrainLodMeshes` with multi-level:

```rust

    (60.0, 70.0),   // LOD 0 <-> LOD 1
    (38.0, 45.0),   // LOD 1 <-> LOD 2
    (18.0, 22.0),   // LOD 2 <-> LOD 3
];
```

### Key Constraints

1. **Cliff wall geometry must match**: After simplifying the terrain surface, the coastline edges change. Cliff walls are generated from land-ocean boundary edges. Either:
   - Generate cliff walls AFTER simplification (re-detect coastline edges on simplified mesh)
   - Or keep cliff walls at full resolution (they're a small fraction of total triangles)

2. **Territory overlay alignment**: The territory overlay mesh (`territory_overlay.rs`) uses the terrain's country map. If the country map changes per-LOD, the overlay must match. Simplest: keep overlay at one resolution and render it as a separate pass.

3. **Beach vertex colors must survive simplification**: The coastal masking BFS, beach factor ramps, and alpha values are baked into vertex colors. QEM with attribute preservation ensures these blend correctly when edges collapse.

4. **Offline only**: All QEM computation happens in `gen_cliff_glb.rs` at build time. No runtime mesh modification needed for the discrete multi-LOD approach. Progressive mesh is the only option that requires runtime mesh updates.
